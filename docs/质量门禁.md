# 质量门禁

质量门禁定义母版如何证明自己可安装、可检查、可测试、可构建、可复制。门禁不是业务质量承诺；它是工程母版的最低可运行证据。

## 命令层级

| 命令 | 适用场景 | 内容 |
| :--- | :--- | :--- |
| `pnpm check` | 日常本地检查 | `lint`、`typecheck`、`test` |
| `pnpm check:full` | CI、根配置、模板、依赖变更后 | `lint`、`typecheck`、`test:coverage`、`build`、`e2e`、`deps:check` |
| `pnpm deps:check` | 依赖治理检查 | `knip --no-config-hints`、`syncpack lint` |
| `pnpm create:experiment ...` | 模板复制验证 | 从 `templates/*` 复制到 `experiments/*` |

`pnpm check` 用于快速确认普通代码变更。`pnpm check:full` 是母版完整门禁，任何影响 root 配置、workspace、模板、共享包、质量工具或脚本的改动都应运行它。

## 大输出命令提醒

会跨包调度、生成 coverage、启动浏览器、构建多个包、安装依赖、检查依赖治理或拉取 CI 日志的命令，都可能产生很长输出。不要把完整输出直接贴进终端对话；默认写入 `tmp/logs/*.log`，再用 `tail` 和 `rg` 摘要检查。

应优先落盘日志的 root 命令包括：`pnpm check:full`、`pnpm check`、`pnpm build`、`pnpm typecheck`、`pnpm test`、`pnpm test:coverage`、`pnpm e2e`、`pnpm deps:check`、`pnpm lint`、`pnpm format`、`pnpm install`、`pnpm exec playwright install --with-deps chromium` 和 `pnpm release`。

应同样谨慎处理的局部命令形态包括：`pnpm --filter <pkg> build`、`pnpm --filter <pkg> typecheck`、`pnpm --filter <pkg> test`、`pnpm --filter <pkg> test:coverage`、`pnpm --filter <pkg> e2e` 和 `pnpm --filter <pkg> pack:check`。`pnpm dev`、`pnpm --filter <pkg> dev`、`pnpm --filter <pkg> preview`、`pnpm --filter <pkg> start`、`pnpm --filter @ts-foundry/ui play` 属于长运行命令，通常放在可见终端或受控后台会话中。

排查 GitHub Actions 时，`gh run watch <run-id> --exit-status`、`gh run view <run-id> --log` 和 `gh run view <run-id> --log-failed` 也应写入 `tmp/logs/`。完整命令表和可复用日志模板见 `GUIDE.md` 的 `Large Command Output` 一节。

## Lint 与格式化

`pnpm lint` 执行 `biome check .`。`pnpm format` 执行 `biome check --write .`。

Biome 管理格式化、基础 lint、import organize 和未使用 import/variable。母版不默认引入 Prettier 或 ESLint。生成文件、构建产物、coverage、Playwright report、test results、`.turbo` 和 `node_modules` 不属于 Biome 源码检查对象。

## TypeScript

`pnpm typecheck` 通过 Turbo 调度各 workspace package 的 `typecheck`。应用包通常执行 `tsc --noEmit`。library、CLI 和 shared packages 也必须通过自身 tsconfig。

TypeScript 检查证明静态类型边界成立，但不替代 Zod 的运行时校验。API、表单和 contracts 的运行时输入输出仍由测试覆盖。

## Vitest

`pnpm test` 执行单元和集成测试。`pnpm test:coverage` 执行同一批测试并生成 v8 coverage。

测试分三类：纯函数和工具测试、schema 和 contracts 测试、组件/API/CLI 集成测试。母版样例的 coverage 数字用于证明链路可运行，不定义复制后业务项目的最终覆盖率门槛。

## Playwright

`pnpm e2e` 通过 Turbo 调度包含 e2e 脚本的包。当前母版覆盖 `apps/web` 和 `templates/react-spa` 的浏览器路径。

e2e 测试使用 Vite dev server 和 MSW，因为前端 hello world 需要验证浏览器中的网络 mock、路由和表单路径。Playwright 不承担所有业务边界验证；它只覆盖关键用户路径。

## Build

`pnpm build` 通过 Turbo 调度需要构建的 workspace package。React 应用执行 TypeScript no-emit 检查和 Vite build。Hono、library、CLI、shared packages 使用 tsdown 输出 ESM 和类型声明。

Vite 可能对前端样例输出 chunk size warning。只要命令退出码为 0，这类提示不是门禁失败。真正的失败标准是非零退出码、类型错误、测试失败、构建失败、e2e 失败或依赖治理失败。

## 依赖门禁

`pnpm deps:check` 执行：

```bash
knip --no-config-hints && syncpack lint
```

Knip 检查 unused dependencies、unlisted dependencies、unresolved imports 和 unused files。配置提示被隐藏，因为母版包含 templates 和 generated artifacts，Knip 的提示会把有效策略误报成建议。

syncpack 检查 package 依赖是否服从 catalog。共享依赖应写成 `catalog:`，具体版本由 `pnpm-workspace.yaml` 统一管理。

## Git hooks

Lefthook 的 pre-commit 阶段只执行 staged files 的 Biome check。pre-push 执行 `pnpm check`。

提交前门禁保持轻量，避免每次 commit 都运行完整浏览器和构建流程。完整验证由 `pnpm check:full` 和 CI 承担。

## CI

GitHub Actions 使用 `actions/checkout`、`pnpm/action-setup`、`actions/setup-node`、Node 24、pnpm frozen lockfile、Playwright Chromium 浏览器安装和 `pnpm check:full`。CI 的职责是证明母版在干净安装环境中可以完成完整门禁。

CI 不执行真实发布，不配置部署，不连接数据库，不启用 Docker。发布和部署属于复制后实例项目的业务工程问题。

## 生成产物规则

以下目录是运行产物，不属于源码对象：

```text
node_modules/
.turbo/
dist/
coverage/
playwright-report/
test-results/
```

这些目录由 `.gitignore` 排除。模板复制脚本也会排除它们，避免把验证产物带入新实验。`public/mockServiceWorker.js` 是 MSW 生成文件，但它是前端模板运行所需文件，因此保留在模板中，同时排除出 Biome 和 Knip 检查。

## 完成声明规则

不能只因为代码看起来合理就声明完成。对母版级改动，完成声明必须有新鲜验证证据。最强证据是最新一次 `pnpm check:full` 退出码为 0，并且 `deps:check` 输出 `No issues found`。

如果只运行了局部门禁，结论必须明确限定范围。例如只运行 `pnpm lint` 时，只能声明格式和 lint 通过，不能声明母版整体通过。
