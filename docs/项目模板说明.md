# 项目模板说明

`templates/` 中的每个目录都是可复制起点。模板不是 README 示例，而是参与母版质量门禁的 workspace package。它们必须能类型检查、测试、构建；前端模板还必须能通过 e2e。

## 样例与模板的区别

`apps/` 放母版自带可运行样例。它们证明母版默认栈可以在真实 workspace 中运行。

`templates/` 放复制源。它们证明以后创建实验时可以从稳定起点开始。`apps/web` 与 `templates/react-spa` 内容接近，是有意设计：前者验证母版应用样例，后者验证复制源。

## 创建实验

```bash
pnpm create:experiment react-spa ui-demo
pnpm create:experiment hono-api api-demo
pnpm create:experiment fullstack contract-demo
pnpm create:experiment node-cli rename-files
pnpm create:experiment ts-lib text-utils
```

脚本会把模板复制到 `experiments/<name>`，并把 package name 改为 `@ts-foundry-experiment/<name>`。复制时会排除 `node_modules`、`dist`、`coverage`、`.turbo`、`playwright-report` 和 `test-results`。

## react-spa

`react-spa` 是纯前端模板。它使用 React、Vite、TanStack Router、TanStack Query、React Hook Form、Zod、Tailwind CSS v4、shadcn/ui、`@ts-foundry/ui`、Motion、MSW、Testing Library 和 Playwright。

hello world 覆盖这些工程路径：

- `/` 和详情页路由由 TanStack Router 定义。
- 列表数据由 TanStack Query 请求。
- 浏览器和测试中的网络响应由 MSW 提供。
- 表单由 React Hook Form 管理，并由 Zod 校验。
- Tailwind 提供基础布局和样式 token。
- shadcn/ui 与 `@ts-foundry/ui` 提供共享基础组件、theme support 和操作反馈。
- Motion 提供有语义的进入或反馈动画。
- Vitest + Testing Library 覆盖组件行为。
- Playwright 覆盖真实浏览器路径。

使用场景：UI demo、前端实验、可视化工具、无后端依赖的 SPA。不要把后端服务、数据库或 CLI 逻辑塞进这个模板。

## hono-api

`hono-api` 是纯后端 API 模板。它使用 Hono、Zod、shared contracts、Vitest 和 tsdown。

hello world 覆盖这些 API 路径：

- `GET /health` 返回健康状态。
- `GET /items` 返回列表。
- `GET /items/:id` 返回详情或 not found。
- `POST /items` 校验 body 并创建对象。
- 错误响应使用固定结构。

使用场景：轻量 API、webhook、小型服务、纯后端计算入口。不要给该模板引入 React、Tailwind、Playwright 或浏览器 mock。

## fullstack-hono-react

`fullstack-hono-react` 是默认 fullstack 模板。它使用 React/Vite 客户端、Hono 服务端、`packages/contracts` 和 `packages/ui`。

hello world 的核心目标是证明前端和后端通过同一份 Zod contract 解释数据边界。后端用 schema 校验输出，前端用 schema parse 响应，测试同时覆盖 UI 和 server handler。

该模板还必须证明 React client 消费 shared UI foundation。shared contracts 与 shared UI package 同时参与 workspace 验证。

使用场景：前端和 API 同仓推进、需要共享 contracts 的 demo、个人 fullstack 实验。该模板不包含数据库、部署、SSR、Next.js 或 NestJS。

## node-cli

`node-cli` 是 Node 命令行工具模板。它使用 Node 24、TypeScript、tsx、tsdown 和 Vitest。

hello world 覆盖这些 CLI 路径：

- 文本输出。
- JSON 输出。
- 参数错误和错误码。
- main 函数测试。
- tsdown 构建 ESM bin。

使用场景：文件处理工具、自动化脚本、开发辅助 CLI、一次性命令工具的稳定化。不要给 CLI 模板引入 React、Hono、Tailwind 或浏览器测试。

## ts-lib

`ts-lib` 是可发布 TypeScript library 模板。它使用 TypeScript、tsdown、Vitest、publint 和 arethetypeswrong。

hello world 覆盖这些包边界：

- `src/index.ts` 导出公共 API。
- Vitest 验证函数行为。
- tsdown 输出 ESM 和声明文件。
- `exports` 和 `types` 字段描述消费入口。
- package check 脚本预留发布边界检查。

使用场景：SDK、纯函数库、共享工具包、未来可能发布的 package。默认只输出 ESM，不提供 CommonJS 兼容层。

## 从实验到稳定目录

实验项目默认留在 `experiments/`。当实验开始被多个项目依赖，或需要长期维护时，按对象类型晋升：

- 成形应用进入 `apps/`。
- 可复用代码进入 `packages/`。
- 可复用起点进入 `templates/`。
- 正式项目进入 `projects/`。

晋升前必须确认依赖方向正确、package name 合理、脚本遵守任务协议，并通过对应质量门禁。

## 模板修改规则

修改模板等同于修改未来项目的默认起点。影响模板依赖、脚本、构建、测试或 e2e 的改动必须运行 `pnpm check:full`。

模板文档和代码必须同步。新增模板时，应同时更新 `pnpm-workspace.yaml`、根脚本需要的配置、`docs/工程治理约定.md`、`docs/技术选型记录.md`、`docs/项目模板说明.md` 和 `AGENTS.md`。
