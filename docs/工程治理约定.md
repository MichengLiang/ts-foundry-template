# 工程治理约定

`ts-foundry-template` 是一个 TypeScript 工程母版，不是一个业务项目。它的职责是把一套固定的工程组织方式、质量门禁、目录边界、任务协议和常见项目模板固化下来，使复制出去的实例工作区不需要重新决定基础工具链。

## 母版对象

本母版服务五类 TypeScript 启动对象：前端应用、后端 API、fullstack 应用、Node CLI、TS library。每一类对象都必须继承同一套工程质量默认值：pnpm workspace、Turborepo、TypeScript strict、Biome、Vitest、Playwright、MSW、Knip、syncpack、Lefthook、Renovate 和 CI。

母版不直接承载业务需求。复制后的实例项目可以根据业务问题决定数据库、部署平台、发布策略、生产监控、鉴权方案、ORM、消息队列和云服务；这些决策不进入母版默认层。

## 两层结构

母版层是 `ts-foundry-template`。它定义目录、配置、脚本、质量门禁、共享包、可运行样例和可复制模板。母版的验收标准是：它自身可以安装、类型检查、测试、构建、运行 e2e，并能复制出符合 workspace 规则的新实验。

实例层是从母版复制出去的具体工作区，例如 `myts` 或未来任何项目。实例层承载真实实验和业务代码。实例可以替换局部技术，但替换是实例自己的选择，不反向改变母版的默认答案。

## 目录边界

```text
apps/         母版自带的可运行样例
packages/     跨应用、跨模板复用的稳定共享包
templates/    可复制的项目起点
experiments/  复制模板后生成的临时或探索性项目
docs/         工程治理、选型、模板和质量门禁文档
scripts/      母版维护脚本
```

`apps/` 不是业务产品目录。它的作用是证明母版中的默认栈能实际运行。`apps/web` 验证 React 前端路径，`apps/api` 验证 Hono API 路径，`apps/fullstack` 验证 React + Hono + shared contracts 路径。

`templates/` 是复制源。模板本身也参与 workspace、typecheck、test、build 和 e2e 检查，因为一个不能被母版验证的模板不是可靠起点。

`experiments/` 是实例工作区的默认落点。新想法、短期验证、小工具、UI demo、API demo、CLI demo 默认先进入这里。只有当一个实验开始承担稳定用途时，才晋升到 `apps/` 或 `packages/`。

## 依赖方向

`apps/*` 可以依赖 `packages/*`。`templates/*` 可以依赖 `packages/*`，因为模板需要展示共享包的使用方式。`experiments/*` 可以依赖 `packages/*`，因为实验应复用稳定能力。

`packages/*` 不依赖 `apps/*`、`templates/*` 或 `experiments/*`。共享包必须保持向下游输出能力，而不是回头读取某个应用或实验的内部实现。

`packages/contracts` 是跨边界协议层，只放 Zod schema、API 输入输出结构和共享类型边界。`packages/ui` 只服务 React 相关项目，并作为 shadcn/ui 驱动的前端 UI foundation，归属基础组件、样式 token、theme support 和 accessible primitives。`packages/utils` 必须保持运行时中立。Node-only 能力放入 `packages/node-utils`。

## 默认值分层

硬默认属于母版身份：pnpm、Turborepo、TypeScript strict、Biome、Vitest、Playwright、MSW、Zod、Knip、syncpack、Lefthook、Renovate、Node 24、ESM。

模板默认属于具体项目类型：React 是前端默认，Hono 是后端默认，TanStack Router 和 TanStack Query 是 React 应用默认，Tailwind CSS v4 与 shadcn/ui 是 UI 样式和组件 foundation 默认，tsdown 是 library 和 CLI 构建默认，Changesets 是 package 发布预留默认。

母版不维护并列默认值。一个默认值如果进入母版，就必须成为当前答案，而不是写成“也可以使用其他工具”。例外选择应出现在实例项目中，并由实例文档记录原因。

## 任务协议

根 `package.json` 是工程控制面板。所有 workspace package 应使用同一组脚本名称：

| 脚本 | 职责 |
| :--- | :--- |
| `dev` | 启动开发服务或开发监听 |
| `build` | 生成可消费产物或生产构建 |
| `typecheck` | 执行 TypeScript 类型检查 |
| `test` | 执行 Vitest 单元或集成测试 |
| `test:coverage` | 执行 Vitest coverage |
| `e2e` | 执行 Playwright 浏览器路径 |
| `clean` | 清理构建和缓存产物 |

缺少某类能力的包可以没有对应脚本，或让 Turbo 不调度该任务；不应发明等价命名，例如把类型检查叫 `verify`，把测试叫 `validate`。统一任务名称是 Turbo 依赖图、CI 和 agent 后续维护的前提。

## 依赖治理

版本来源集中在 `pnpm-workspace.yaml` 的 catalog。workspace package 中的共享依赖应使用 `catalog:`，避免每个模板和样例复制版本号。

`syncpack lint` 检查依赖版本是否服从 catalog。`knip` 检查未使用依赖、未列出的依赖、未解析的导入和未使用文件。生成产物如 `coverage/`、`dist/`、`.turbo/`、`playwright-report/`、`test-results/` 不属于源码治理对象。

新增依赖必须满足一个明确位置：根部 devDependency 只放全仓工具；应用依赖只放该应用运行需要；模板依赖只放复制后项目需要；共享包依赖只放该包的公共能力需要。

## 实验创建与晋升

新实验通过模板复制创建：

```bash
pnpm create:experiment react-spa ui-demo
pnpm create:experiment hono-api api-demo
pnpm create:experiment fullstack contract-demo
pnpm create:experiment node-cli rename-files
pnpm create:experiment ts-lib text-utils
```

复制脚本只复制源码、配置和模板必需文件；不会复制 `node_modules`、`dist`、`coverage`、`.turbo`、`playwright-report` 或 `test-results`。MSW worker 属于前端模板运行文件，会保留在对应 `public/` 目录。

实验晋升时必须确认三件事：它的用途已经稳定，它的依赖边界没有污染共享包，它能通过目标层级的质量门禁。晋升到 `apps/` 表示它是一个成形应用；晋升到 `packages/` 表示它提供可复用 API。

## 新工具准入

新增默认工具必须同时满足三个条件：减少重复决策，能被自动验证，能跨至少两类项目复用。只服务单个业务场景的工具进入实例项目；只服务单个模板类型的工具进入该模板；只有跨项目的工程质量能力才进入母版根部。

默认不引入 Prettier、ESLint、Jest、Webpack、Nx、Nest、Next、数据库 ORM、Docker 或部署平台。这个排除不是永久禁令，而是母版对象边界。实例项目需要这些工具时，由实例自己引入并记录原因。

## Agent 维护规则

未来 agent 修改本仓库时，应优先保持现有母版结构。新增项目默认进入 `experiments/`，稳定应用进入 `apps/`，共享代码进入 `packages/`，复制源进入 `templates/`。不要把一次性文件放到根目录，不要让纯后端或 CLI 继承 React/Tailwind 依赖，不要绕过 pnpm、Biome、Vitest 和 Turbo 的统一入口。

完成任何影响工程行为的改动后，必须运行对应验证。影响根配置、依赖、模板、共享包或脚本时，至少运行 `pnpm check:full`。
