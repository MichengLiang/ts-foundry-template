# 技术选型记录

本文记录 `ts-foundry-template` 的默认技术答案。它不是工具百科，也不维护多条并列路线。母版只保存当前对象边界内的固定选择、选择理由和明确排除项。

## pnpm workspace + catalog

pnpm 是包管理默认值。它负责 workspace、lockfile、workspace 协议、catalog 版本集中管理和可重复安装。母版使用 pnpm 10，并在 `package.json` 中固定 `packageManager`。

选择 pnpm 的理由是：它的 workspace 能力成熟，catalog 能把 React、TypeScript、Vitest、Biome、Vite、Tailwind、TanStack 等共享版本集中到一个位置，适合长期维护的单仓母版。母版不同时支持 npm、Yarn 或 Bun install，因为多包管理器会稀释 lockfile 和依赖治理。

## Turborepo

Turborepo 是任务编排默认值。它负责跨包任务图、并行执行、缓存和上游依赖构建顺序。母版的任务协议围绕 `build`、`typecheck`、`test`、`test:coverage`、`e2e` 和 `clean` 展开。

选择 Turborepo 的理由是：它足够轻，适合一个个人长期 workbench，不要求引入大型项目图和生成器体系。母版不默认使用 Nx，因为 Nx 的治理能力更重，适合另一类组织级产品仓，而不是当前的实验母版。

## TypeScript 6 strict

TypeScript 是边界权威。母版启用 strict、`noUncheckedIndexedAccess`、`exactOptionalPropertyTypes`、`noImplicitOverride`、`verbatimModuleSyntax` 等严格选项，并通过 `packages/tsconfig` 提供 `base`、`react`、`node`、`lib`、`test` 分环境配置。

应用包通常 `noEmit`，由 Vite 或 tsdown 负责构建。library 和 CLI 包通过 tsdown 输出 ESM 和声明文件。跨包 API 不能只靠 README 约定，必须由导出的类型、Zod schema 和测试共同固定。

## Biome

Biome 是全仓格式化和 lint 默认值。它负责格式化、基础 lint、import organize 和常见可疑代码检查。

选择 Biome 的理由是：它速度快、配置少、职责集中，适合单仓统一治理。母版不默认引入 Prettier，也不默认引入 ESLint。实例项目如果确实需要框架专属 ESLint 规则，可以在实例层添加，并在实例文档中记录原因。

## Vitest + v8 coverage

Vitest 是单元测试和集成测试默认值。它和 Vite、ESM、TypeScript 的组合自然，适合前端组件、纯函数、schema、Hono handler、CLI main 函数和共享包测试。

coverage 使用 v8 provider。母版中的 coverage 主要证明测试链路可运行，不把 hello world 的覆盖率数字定义成业务项目承诺。复制后的稳定项目可以按自身风险提高覆盖率要求。

## Playwright

Playwright 是浏览器 e2e 默认值。母版只用它覆盖关键用户路径，不把所有边界都推给浏览器测试。

选择 Playwright 的理由是：它的浏览器自动化能力稳定，适合验证 React/Vite 应用真实页面行为。母版不默认使用 Cypress，因为 Playwright 已经满足当前 e2e 目标，且减少并列测试栈可以降低维护成本。

## MSW + faker

MSW 是网络边界 mock 默认值。它用于前端测试和浏览器路径中的 API 响应模拟，使 TanStack Query、路由、表单提交和网络请求在测试中保持真实调用形态。

faker 是测试数据默认值。测试数据必须 seed，避免随机数据造成不可复现失败。MSW 不应替代业务 fixture，也不应把后端逻辑复制进前端测试；它只模拟网络边界。

## Zod contracts

Zod 是运行时边界校验默认值。它服务 API request/response、表单输入、shared contracts 和测试 fixture。

母版选择 Zod contracts，而不是把所有通信绑定到某个全栈 RPC 框架，是因为当前对象同时包含纯后端、纯前端、CLI、library 和 fullstack。显式 schema 更适合作为跨项目边界，且可以被前后端共同解析。

## React 19 + Vite 8

React 是前端 UI runtime 默认值。Vite 是前端开发和构建默认值。`apps/web` 和 `templates/react-spa` 用它们验证 SPA 路径。

选择 React + Vite 的理由是：React 生态最大，Vite 对现代 ESM/TS 前端开发足够直接。母版不默认使用 Webpack，也不把 Next.js 作为前端或 fullstack 根默认，因为当前母版不是 SSR 产品框架模板。

## TanStack Router + TanStack Query

TanStack Router 是 React 应用路由默认值。它提供类型化路由、路由参数和 search 边界。TanStack Query 是 server state 默认值。它管理异步读取、缓存、失效和加载状态。

选择这组工具的理由是：它们把路由边界和服务端状态边界显式化，适合 TypeScript 工程质量优先的前端模板。组件不应直接散落请求生命周期，路由参数也不应只靠字符串约定。

## React Hook Form + Zod

React Hook Form 是表单状态默认值。Zod 负责表单输入校验和类型推导。模板中的表单 hello world 用它们证明输入、校验、错误展示和提交路径可以组合运行。

母版不引入更重的表单框架，因为当前默认对象只需要覆盖常见表单状态和 schema 校验，不需要把业务表单建模系统化。

## Tailwind CSS v4 + Motion

Tailwind CSS v4 是 UI 样式默认值。它负责 token、布局和组件样式的基础表达。Motion 是有语义交互动画默认值，用于进入、反馈和布局变化，不作为装饰动画默认铺开。

选择 Tailwind 的理由是：它能让小实验快速获得一致样式能力，也能被 React 模板直接验证。Motion 只进入前端和 fullstack 模板，不进入纯后端、CLI 或 library。

## Hono

Hono 是后端 API 默认值。它使用 Web Standard Request/Response，适合 Node 24 下的轻量 API、webhook、小服务和 fullstack 实验。

母版不默认使用 NestJS，因为 Nest 的 DI、装饰器和应用框架约束会增加实验成本。Fastify 是成熟后端框架，但在当前母版中不是默认值；当实例项目需要 Fastify 插件生态或特定性能形态时，由实例项目自行选择。

## fullstack-hono-react

母版的 fullstack 默认是 React/Vite 前端 + Hono API + shared contracts。它强调前后端边界清楚，前端和后端通过 `packages/contracts` 共享 Zod schema。

母版不把 Next.js、TanStack Start 或其他 SSR 框架作为默认 fullstack 路径。SSR、文件路由、部署一体化属于另一类产品框架问题，不属于当前第一母版对象。

## tsdown + ESM

tsdown 是 library 和 CLI 构建默认值。它负责输出现代 ESM、类型声明和 CLI bin 入口产物。

母版不默认提供 CommonJS 兼容层。Node 24 和现代 bundler 都能消费 ESM；如果某个实例项目确实需要 CJS，那是该实例的兼容性需求，不进入母版默认层。

## Knip + syncpack + Renovate

Knip 检查未使用依赖、未列出的依赖、未解析导入和未使用文件。syncpack 检查依赖版本是否服从 catalog。Renovate 负责依赖升级分组。

这三者解决不同层面的依赖治理：Knip 处理“是否还需要”，syncpack 处理“版本是否一致”，Renovate 处理“如何持续升级”。它们共同减少以后重复追版本、查废依赖和修 lockfile 漂移的成本。

## Lefthook + GitHub Actions

Lefthook 是本地 Git hooks 默认值。pre-commit 只跑快速 Biome staged check，pre-push 跑 `pnpm check`。GitHub Actions 是 CI 默认值，执行 frozen install 和 `pnpm check:full`。

本地 hooks 不承担所有验证，因为过重的 commit 门禁会拖慢实验节奏。完整门禁放到 `check:full` 和 CI，确保母版在关键边界上可验证。

## Changesets

Changesets 是 package 版本和发布预留默认值。母版不启用真实发布 workflow，但保留 `changeset` 和 `release` 命令，使可发布 packages 在需要时有明确路径。

发布策略不属于母版核心问题。它在母版中只作为 package 版本管理能力存在，不绑定 npm 发布、私有 registry 或 CI release。

## 明确非目标

数据库、ORM、迁移工具、部署平台、Docker、生产发布、云服务、SSR 框架、Next.js、NestJS、远程 Turbo cache 和真实 npm 发布 workflow 不属于当前母版对象。它们都是实例项目或后续独立模板的对象，不进入本母版默认集合。
